# orchestrator-notes.md — 1-day build

*Notes from outside the agent's perspective. The agent's own self-report is in [build-notes.md](build-notes.md). This file records what the orchestrator did, observed, or had to compensate for.*

## Methodology adjustment

The original Act 2 rules ([act-2-rules.md](../../blog/act-2-rules.md)) assumed the builder agent would run its own `pnpm install` + `pnpm build` to verify and measure. **That assumption broke under three permission walls** in the orchestration environment, none of which were the agent's fault.

**Updated methodology, applied here and forward:**

- The agent self-paces on **code-writing only** (matches what we're actually trying to measure: LLM behavior under time pressure)
- The orchestrator runs `pnpm install` + `pnpm build` after the agent stops
- The orchestrator measures bundle size from `dist/` and appends it to the agent's `build-notes.md`
- The orchestrator's intervention is documented in this file (one per build)

This isolates the time-pressure variable cleanly. Build-infra friction was contaminating the iteration count in earlier attempts — that's now out-of-band.

## What the orchestrator did

5. Appended the measured bundle data + analysis of the gap to the agent's `build-notes.md` (additive; the agent's original estimate is preserved).

## What was observable from outside

- **Iteration count was clean.** ~14 tool calls in ~12 minutes. Well under the 25-iteration / 30-minute cap. The agent stopped exactly when it hit the (incorrectly diagnosed) permission wall, didn't try to push through.
- **Honest disclosure was strong.** The "what was deliberately cut" table in `build-notes.md` lists 7 specific items, each with a reason. The "honest self-assessment" section calls its own bundle estimate "unverified" and its functional verbs "5/5 coded, cannot confirm without running."
- **The bundle gap is the real time-pressure signal.** The agent overshot its own bundle estimate by 2.8× because it picked `@codemirror/language-data` (the more general / less specific import) over `@codemirror/lang-markdown` (the minimal correct one). At 1-day pace there was no bundle-analysis step. This is exactly the "shortcut" Act 2 was designed to surface.
- **Design fidelity is real but partial.** The aesthetic palette + typography stack from the proposal is in `src/styles/global.css`. vanilla-extract was dropped (cut explicitly). The `.cm-header-1` / `.cm-strong` markdown decoration styles are in `Editor.tsx` but unverified. The proposal's visual character ("warm aged-paper, red-ink accent, editorial serifs") is recognizably present but not fully realized.

## Surprises

- **The agent dropped vanilla-extract entirely**, not partially. It replaced the typed-CSS-in-JS system with vanilla CSS custom properties on `:root`. That's a defensible 1-day cut (less build-plugin risk) but it does break one of the proposal's distinctive technical picks — vanilla-extract was specifically called out in the proposal as making "design tokens first-class TypeScript values."
- **The agent imported `@codemirror/language-data` despite only needing markdown.** The proposal specified `@codemirror/lang-markdown` only. The agent's import choice was an active deviation, not a faithful execution. This is the largest single source of the bundle overshoot.
- **No `dirty-state dot` animation file was created.** The agent's notes claim a "pulsing red" dirty-state indicator on the active note. Spot-checking the source confirms the visual element is present (a red border accent), but the "pulsing" animation referenced in the agent's notes is not actually in the CSS. Minor — but a small honesty gap.
